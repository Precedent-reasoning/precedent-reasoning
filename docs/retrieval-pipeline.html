§ PRECEDENT — RETRIEVAL PIPELINE

How a plain-English question becomes five cited cases

Eight stages, spanning agent.py, tools/corpus.py and ingest.py's local index — one embedding, two parallel search strategies, one rerank, one fetch. No step ever touches the internet; everything queries the local index built by ingest.py.

01 User query

A natural-language description of the user's legal situation.

"my landlord won't return my bond after I moved out"
agent.py — situation text
02 Embed the query

Prefixed with search_query: and encoded into a single dense vector that captures meaning, not just keywords.

nomic-embed-text-v1.5 · 768-dim · MPS / bfloat16
semantic
03a Vector search

Cosine similarity over every chunk's embedding — finds chunks that mean the same thing, even with different wording.

LanceDB · top 50 chunks
lexical
03b Full-text search

BM25 keyword search over the same chunk text — catches exact citations, party names, statutory terms embeddings can blur.

LanceDB FTS · top 50 chunks
04 Merge & dedupe

Both result sets are combined by chunk_id; a chunk found by both searches is kept once.

tools/corpus.py — search_cases() · ~50–100 candidate chunks
candidates
05 Rerank

A cross-encoder scores the query and each candidate together — slower than the first pass, but far more precise, so it only needs to run on the short candidate list.

zerank-1-small-reranker · cross-encoder
sorted by relevance
06 Select top cases

Highest-scoring chunk per case wins the case slot; the case list is capped at max_results (default 5). Its top 3 scored chunks are remembered, keyed by URL, for the fetch step below.

up to 5 case_id / url pairs · top-3 chunk_index per case cached
case_id, url
persisted store
07 Fetch case text

A flat head-of-document read would often miss the passage that made a long judgment match in the first place. Instead, LanceDB is queried for that case's actually-matched chunks (from stage 06) plus its first and last chunk — catchwords and orders, wherever the match fell. Falls back to a capped SQLite read if the case wasn't matched by a recent search.

LanceDB chunks (matched + first/last) → SQLite fallback (≤8,000 chars) · fetch_case() / fetch_cases()
up to 5 cases, matched passages + bookends
